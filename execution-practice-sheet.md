# Codex CLI 첫 실습 진행 매뉴얼

이 문서는 진행자가 화면을 공유하고, 참가자 3명이 각자 자기 노트북에서 그대로 따라 하는 2시간 실습 매뉴얼이다. 참가자는 터미널을 처음 써도 된다는 전제로 작성했다.

오늘의 목표는 멋진 웹사이트를 완성하는 것이 아니다. 자기 주제로 작은 정적 웹 미니 프로젝트를 만들고, Codex CLI에게 계획, 구현, 확인, 테스트, GitHub 업로드까지 맡겨보는 것이다.

## 오늘 완성할 것

수업이 끝나면 각 참가자 컴퓨터에 아래와 같은 폴더가 생긴다.

```text
my-codex-project/
  README.md
  index.html
  style.css
  script.js
  test.js
```

각 파일의 의미는 수업 중 Codex가 설명한다.

- `README.md`: 프로젝트가 무엇인지 설명하는 문서
- `index.html`: 브라우저에 보이는 화면 뼈대
- `style.css`: 색상, 간격, 글자 크기 같은 디자인
- `script.js`: 버튼이나 입력창이 움직이게 하는 동작
- `test.js`: 프로젝트가 제대로 만들어졌는지 확인하는 간단한 검사 파일

## 수업 규칙

- 참가자는 코드를 직접 고치지 않는다. 파일 생성과 수정은 Codex에게 맡긴다.
- 참가자는 Codex에게 무엇을 원하는지 말하고, 결과를 확인한다.
- 막히면 혼자 오래 붙잡지 않는다. 3분 이상 같은 화면에서 멈추면 손을 든다.
- 설치가 한 사람만 오래 막히면 그 사람은 잠시 진행자 화면을 보며 따라오고, 쉬는 시간이나 마지막에 따로 해결한다.
- 오늘은 백엔드, 로그인, 결제, 데이터베이스, 외부 API를 쓰지 않는다.
- 실제 개인정보, 비밀번호, 회사 내부 자료, API 키는 입력하지 않는다.
- 명령어를 외우는 수업이 아니다. “무엇을 하려고 이 도구를 쓰는지”를 이해하는 수업이다.

## 전체 시간표

| 시간 | 할 일 | 목표 |
|---|---|---|
| 0:00-0:05 | 수업 목표와 규칙 확인 | 오늘 만들 범위 이해 |
| 0:05-0:15 | 자유 주제 정하기 | 자기 프로젝트 방향 정하기 |
| 0:15-0:35 | Node.js 확인과 설치 | Codex CLI를 설치할 준비 |
| 0:35-0:50 | Codex CLI 설치와 로그인 | Codex 실행 성공 |
| 0:50-1:05 | 작업 폴더 만들기와 Plan Mode 시작 | 바로 만들지 않고 먼저 계획 |
| 1:05-1:15 | 계획 검토와 수정 요청 | 사람이 구조를 승인 |
| 1:15-1:30 | Codex가 미니 프로젝트 구현 | 파일 생성 |
| 1:30-1:40 | Node 로컬 서버로 브라우저 확인 | 화면 열기 |
| 1:40-1:52 | 수동 확인과 test.js 실행 | 검증 체험 |
| 1:52-2:00 | GitHub 저장소 연결과 업로드 | 결과 공유 |

진행자는 시간을 엄격히 본다. 마지막 GitHub 단계는 압축 단계다. 시간이 부족하면 Codex에게 “다음에 이어서 업로드할 요약”을 만들게 하고 종료한다.

## 0단계: 진행자 오프닝

진행자가 읽는다.

> 오늘은 AI에게 바로 “만들어줘”라고 하지 않습니다. 먼저 계획을 세우고, 그 계획을 사람이 검토한 뒤, Codex에게 구현을 맡깁니다. 설치부터 GitHub 업로드까지 한 번에 해보지만, 핵심은 명령어 암기가 아니라 AI와 일하는 순서입니다.

참가자가 준비할 것은 노트북과 인터넷뿐이다. 사전 준비가 없어도 시작한다.

## 1단계: 내 자유 주제 정하기

시간: 0:05-0:15

진행자가 말한다.

> 각자 만들고 싶은 주제를 하나 정합니다. 단, 오늘은 전체 서비스를 만들지 않습니다. 내 주제를 보여주는 작은 웹 미니 프로젝트만 만듭니다.

참가자는 종이나 메모장에 한 문장으로 적는다.

예시:

- 내가 읽은 책을 추천하는 작은 페이지
- 내 운동 목표를 보여주는 체크 페이지
- 우리 가게 메뉴를 소개하는 페이지
- 여행 후보지를 고르는 추천 페이지
- 공부 계획을 보여주는 페이지
- 오늘의 기분에 따라 문구를 보여주는 페이지

진행자가 다시 말한다.

> 주제는 자유입니다. 하지만 오늘 만들 기능은 버튼 또는 입력창 하나입니다. 버튼을 누르거나 입력하면 화면 내용이 바뀌면 충분합니다.

진행자는 참가자 주제가 너무 크면 즉시 줄인다.

- “쇼핑몰”은 “상품 3개를 소개하는 페이지”로 줄인다.
- “예약 시스템”은 “예약 문의 버튼을 누르면 안내 문구가 나오는 페이지”로 줄인다.
- “AI 추천 서비스”는 “버튼을 누르면 미리 정한 추천 문구가 바뀌는 페이지”로 줄인다.
- “회원 관리”는 “회원 기능 없이 소개 화면만 만드는 것”으로 줄인다.

## 2단계: Node.js가 있는지 확인하기

시간: 0:15-0:20

진행자가 말한다.

> Node.js는 JavaScript를 내 컴퓨터에서 실행하게 해주는 프로그램입니다. npm은 Node.js와 같이 설치되는 도구 상자입니다. 우리는 npm으로 Codex CLI를 설치하고, 나중에 만든 웹 프로젝트도 확인합니다.

참가자 행동:

1. Windows 사용자는 시작 메뉴에서 PowerShell을 연다.
2. macOS 사용자는 Spotlight 검색에서 Terminal을 연다.
3. 진행자 화면을 보고 Node.js 버전 확인 명령을 따라 입력한다.
4. 이어서 npm 버전 확인 명령도 따라 입력한다.

성공 화면:

- 숫자와 점이 섞인 버전이 나오면 성공이다.
- 예: `v`로 시작하는 Node 버전, 숫자로 된 npm 버전

실패 화면:

- “명령을 찾을 수 없음”
- “인식되지 않는 명령”
- 아무 반응이 없음

진행자가 말한다.

> 버전이 나오면 설치를 건너뜁니다. 버전이 안 나오면 지금 Node.js를 설치합니다.

## 3단계: Node.js 설치하기

시간: 0:20-0:35

Node.js가 이미 설치된 사람은 이 단계에서 옆 사람을 돕거나 자기 주제를 더 작게 정리한다.

참가자 행동:

1. 브라우저를 연다.
2. 구글에서 `nodejs`라고 검색한다.
3. 공식 Node.js 사이트에 들어간다.
4. LTS라고 표시된 안정 버전을 선택한다.
5. 자기 OS에 맞는 설치 파일을 다운로드한다.
6. 다운로드한 설치 파일을 실행한다.
7. 설치 화면에서는 특별한 이유가 없으면 기본값으로 계속 진행한다.
8. 설치가 끝나면 PowerShell 또는 Terminal을 완전히 닫는다.
9. 새 PowerShell 또는 Terminal을 다시 연다.
10. 진행자 화면을 보고 Node.js와 npm 버전 확인을 다시 한다.

진행자 멘트:

> 설치 후에는 터미널을 닫았다가 다시 열어야 새 프로그램을 인식하는 경우가 많습니다. 설치했는데도 버전이 안 나오면 먼저 터미널을 새로 열어봅니다.

막혔을 때:

- 다운로드 사이트가 헷갈리면 진행자 화면의 공식 사이트 주소를 보고 확인한다.
- 설치 중 관리자 권한 확인이 나오면 본인 컴퓨터라면 허용한다.
- 회사나 학교 컴퓨터라 설치가 막히면 오늘은 진행자 화면을 보며 실습 흐름을 따라간다.

## 4단계: Codex CLI 설치하기

시간: 0:35-0:43

진행자가 말한다.

> 이제 Codex CLI를 설치합니다. Codex CLI는 터미널에서 실행하는 AI 개발 도구입니다. 설치 명령은 바뀔 수 있으니 수업 당일에는 공식 문서를 먼저 확인하고, 진행자 화면에 나온 명령을 그대로 따라 합니다.

참가자 행동:

1. 진행자가 공식 Codex 설치 문서를 화면에 띄운다.
2. 참가자는 진행자 화면을 보고 설치 명령을 복사한다.
3. PowerShell 또는 Terminal에 붙여넣고 실행한다.
4. 설치가 끝날 때까지 기다린다.
5. 진행자 화면을 보고 Codex 버전 확인을 따라 한다.

성공 화면:

- Codex 버전이 나오면 성공이다.

실패 화면:

- 권한 오류
- npm 오류
- 네트워크 오류
- 명령을 찾을 수 없음

진행자가 말한다.

> 오류가 나오면 그 내용을 지우지 말고 그대로 둡니다. 잠시 뒤 Codex를 실행할 수 있는 사람부터 진행하고, 막힌 사람은 진행자가 오류를 봐줍니다.

막혔을 때 진행자 처리:

- Node.js 버전이 안 나오면 Node 설치 단계로 돌아간다.
- npm 관련 오류면 터미널을 새로 열고 다시 시도한다.
- 권한 오류면 Windows는 PowerShell을 관리자 권한으로 열지 말고, 먼저 일반 PowerShell에서 다시 시도한다. macOS는 설치 문서의 권장 방법을 다시 확인한다.
- 5분 이상 해결이 안 되면 그 참가자는 진행자 화면을 보며 Plan Mode와 프롬프트 작성만 함께 한다.

## 5단계: Codex CLI 로그인하기

시간: 0:43-0:50

진행자가 말한다.

> 오늘은 ChatGPT 계정으로 로그인합니다. API key는 쓰지 않습니다. 브라우저가 열리면 ChatGPT 계정으로 로그인하고 승인합니다.

참가자 행동:

1. PowerShell 또는 Terminal에서 Codex를 실행한다.
2. 브라우저가 열리면 ChatGPT 계정으로 로그인한다.
3. 승인 또는 Continue 버튼이 나오면 누른다.
4. 다시 터미널로 돌아온다.
5. Codex 대화 화면이 보이면 성공이다.

성공 기준:

- 터미널 안에 Codex와 대화할 수 있는 화면이 열린다.
- 프롬프트를 입력할 수 있다.

로그인 실패 시 참가자가 진행자에게 보여줄 것:

- 브라우저 화면
- 터미널에 남은 오류 문구
- Windows인지 macOS인지

진행자 대처 멘트:

> 로그인 문제는 계정, 브라우저, 터미널 연결 문제일 수 있습니다. 오늘 핵심은 흐름을 익히는 것이므로, 로그인에 오래 막히면 진행자 화면을 보며 같은 문장을 따라 적고 마지막에 다시 시도합니다.

## 6단계: 작업 폴더 만들기

시간: 0:50-0:55

진행자가 말한다.

> 이제 오늘 프로젝트를 넣을 폴더를 만듭니다. 폴더 이름은 영어와 숫자만 쓰겠습니다. 한글이나 공백이 들어가면 처음 실습에서 헷갈릴 수 있습니다.

참가자 행동:

1. 바탕화면이나 문서 폴더 안에 새 폴더를 만든다.
2. 폴더 이름은 `my-codex-project`처럼 단순하게 한다.
3. 터미널에서 그 폴더로 이동한다.

진행자 안내 방식:

- Windows 참가자에게는 파일 탐색기에서 폴더를 열고, 주소창을 이용해 터미널을 여는 방법을 보여준다.
- macOS 참가자에게는 Finder에서 폴더를 만들고 Terminal에서 그 위치로 이동하는 방법을 보여준다.
- 이동이 어려운 참가자는 진행자가 옆에서 도와준다.

확인:

- 터미널 현재 위치가 방금 만든 폴더인지 확인한다.
- 폴더 안이 비어 있으면 좋다.

진행자 멘트:

> 지금부터 Codex는 이 폴더 안에서만 작업합니다. 그래서 폴더 위치가 중요합니다.

## 7단계: Plan Mode로 시작하기

시간: 0:55-1:05

진행자가 말한다.

> 지금부터는 만들지 않습니다. 먼저 계획만 세웁니다. 이 단계를 Plan Mode라고 부르겠습니다. Codex에서 Plan Mode를 켤 수 있으면 켜고, 동시에 문장으로도 “아직 만들지 말고 계획만 세워줘”라고 분명히 말합니다.

참가자 행동:

1. Codex 화면에서 Plan Mode를 켤 수 있으면 진행자 안내에 따라 켠다.
2. 아래 문장을 자기 주제에 맞게 바꿔서 Codex에 보낸다.

참가자가 Codex에 보낼 문장:

```text
Plan Mode로 진행하자.

아직 파일을 만들거나 수정하지 말고, 먼저 계획만 세워줘.

나는 내 자유 주제로 정적 웹 미니 프로젝트를 만들 거야.

내 주제:
[여기에 내 주제를 쓴다]

조건:
- 파일은 README.md, index.html, style.css, script.js, test.js로 구성한다.
- 백엔드, 로그인, 데이터베이스, 결제, 외부 API는 쓰지 않는다.
- 버튼 또는 입력창 중 하나만 사용한다.
- 버튼을 누르거나 입력하면 화면 내용이 바뀌어야 한다.
- 초보자가 오늘 안에 만들 수 있을 만큼 작아야 한다.
- Node.js 로컬 서버로 브라우저에서 확인할 수 있어야 한다.
- test.js는 Node.js 기본 기능만 사용해서 프로젝트 파일과 연결 상태를 확인해야 한다.

계획에는 다음을 포함해줘.
1. 프로젝트 이름
2. 대상 사용자
3. 화면 구성
4. 파일별 역할
5. 버튼 또는 입력창 동작
6. 구현 순서
7. 브라우저 확인 방법
8. 테스트 방법
9. GitHub에 올릴 때 확인할 것
```

Codex가 바로 파일을 만들려고 하면 참가자가 다시 보낸다.

```text
아직 구현하지 마.
지금은 계획만 다시 써줘.
```

진행자 확인:

- 참가자 화면에 계획이 나왔는가?
- 계획에 파일 5개가 들어갔는가?
- 기능이 버튼 또는 입력창 하나로 제한됐는가?
- 너무 큰 기능이 들어가지 않았는가?

## 8단계: 계획을 검토하고 한 번 수정 요청하기

시간: 1:05-1:15

진행자가 말한다.

- 내가 이해하지 못하는 어려운 기능이 들어가 있지 않은가?

주제가 잘 안 드러나는 경우:


테스트가 약한 경우:


초보자에게 어려운 기능이 들어간 경우:

```text

백엔드, 로그인, 데이터베이스, 외부 API 없이 정적 웹 프로젝트로 다시 줄여줘.
```

계획이 마음에 들면 최종 승인한다.

```text
이 계획으로 진행하자.
이제 파일을 만들어줘.
구현이 끝나면 파일별로 무엇을 만들었는지 설명하고, 내가 브라우저에서 확인할 순서를 알려줘.
```
