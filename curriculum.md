# 개인 목표 기반 AI 바이브코딩 실습 커리큘럼

## 목적

3명의 참가자가 각자 다른 목표, OS, AI 도구, 모델을 쓰더라도 같은 AI 협업 루틴을 경험한다.

첫 2시간의 성공 기준은 같은 앱을 완성하는 것이 아니다. 각자 자기 목표를 작게 분해하고, AI에게 컨텍스트를 주고, 결과를 검증하고, 문제를 좁혀 수정하고, 보안 위험을 확인하는 것이다.

## 참가자 전제

- 참가자는 각자 만들고 싶은 것, 자동화하고 싶은 것, 정리하고 싶은 문제 중 하나를 가져온다.
- 사용 도구는 통일하지 않는다.
- 로컬 개발이 어려운 참가자는 웹 기반 AI 도구나 노코드/로우코드 도구로 진행한다.
- 코드 프로젝트가 아닌 목표도 허용한다.

## 2시간 진행 흐름

| 시간 | 활동 | 핵심 실습 |
|---|---|---|
| 0:00-0:10 | 각자 목표를 한 문장으로 적는다. | AI에게 맡기기 전에 사람이 목표를 정의한다. |
| 0:10-0:25 | 목표를 `앱`, `자동화`, `문서/콘텐츠`, `데이터 정리`, `학습/실험` 중 하나로 분류한다. | 작업 유형에 맞는 최소 산출물을 정한다. |
| 0:25-0:40 | `사용자`, `입력`, `출력`, `데이터`, `예외 상황`을 적는다. | 데이터, 구조, 예외를 먼저 설계한다. |
| 0:40-1:05 | 각자 가진 AI 도구와 OS에서 가장 작은 첫 결과물을 만든다. | 도구를 목적에 맞게 사용한다. |
| 1:05-1:20 | AI에게 결과물 구조를 설명시키고 본인이 이해한 내용을 적는다. | 설명 가능한 결과만 받아들인다. |
| 1:20-1:35 | 기능 하나를 추가하거나 수정하기 전 버전 저장 지점을 만든다. | 정상 상태를 저장하고 되돌릴 준비를 한다. |
| 1:35-1:50 | 요구 변경 또는 오류 상황을 만들고 원인을 좁혀 수정한다. | 에러 복붙 대신 의심 범위를 좁혀 AI에게 지시한다. |
| 1:50-2:00 | 보안, 개인정보, API 키, 공유 위험을 확인하고 다음 작업을 정리한다. | 보안과 컨텍스트 관리를 작업 루틴에 넣는다. |

## 목표 유형별 최소 산출물

| 목표 유형 | 최소 산출물 | 예시 |
|---|---|---|
| 앱 | 클릭하거나 입력하면 한 가지 동작이 되는 화면 | 예약 요청 폼, 개인 포트폴리오 초안, 계산 화면 |
| 자동화 | 파일, 메일, 표, 폴더, 텍스트 중 하나를 변환하거나 정리하는 흐름 | CSV 정리, 폴더명 일괄 변경, 메일 초안 생성 |
| 문서/콘텐츠 | AI와 함께 만든 초안, 검토 기준, 수정 로그 | 소개서, 강의안, 블로그 글, 제안서 |
| 데이터 정리 | 원본 데이터, 정리 규칙, 변환 결과 | 엑셀 표 정리, 중복 제거, 카테고리 분류 |
| 학습/실험 | 질문 목록, 실험 프롬프트, 결과 비교표 | 모델별 답변 비교, 공부 계획 생성, 리서치 실험 |

## OS와 도구 차이 처리

### Windows

- 로컬 개발 가능: VS Code, Cursor, Windsurf, Claude Code, Codex 등으로 진행한다.
- 로컬 개발이 어려움: ChatGPT, Claude, Gemini, Replit, Bolt, Lovable 같은 웹 기반 도구로 진행한다.
- 버전 저장 대체: git, OneDrive 버전 기록, 파일 복사본, 도구 내 히스토리 중 하나를 사용한다.

### macOS

- 로컬 개발 가능: VS Code, Cursor, Windsurf, 터미널 기반 에이전트 도구로 진행한다.
- 로컬 개발이 어려움: 웹 기반 AI 도구로 진행한다.
- 버전 저장 대체: git, Time Machine, 파일 복사본, 도구 내 히스토리 중 하나를 사용한다.

### 웹 기반 도구만 사용하는 경우

- 코드 실행보다 결과물과 작업 로그에 집중한다.
- 버전 저장은 복제본 만들기, 히스토리 저장, 중간 결과 복사로 대체한다.
- 보안 점검은 공개 공유 링크, 입력한 개인정보, API 키 노출 여부를 중심으로 한다.

## 모델 차이 처리

모델별 품질을 비교하지 않는다. 같은 목표를 두고 다음 차이를 비교한다.

- 어떤 컨텍스트를 줬을 때 답이 좋아졌는가?
- 어떤 제약을 빠뜨렸을 때 결과가 흔들렸는가?
- 어떤 검증 질문을 했을 때 오류가 발견됐는가?
- 긴 대화를 새 대화로 옮겼을 때 결과가 안정됐는가?

## 핵심 실습 루틴

| 영역 | 좋은 작업 습관 | 수업에서의 체험 |
|---|---|---|
| 도구 선택 | 목적에 맞는 AI 도구와 작업 환경을 고른다. | 각자 가진 도구로 진행하되, 한 번은 에이전트형 도구 화면을 공유 데모로 본다. |
| 프롬프트 | 데이터, 구조, 예외를 짧게 설계한 뒤 작은 단위로 요청한다. | 모든 목표를 `입력`, `출력`, `제약`, `예외`로 분해한다. |
| 외부 컨텍스트 | 필요한 외부 정보, 파일, 도구 접근 범위를 먼저 정한다. | 공통은 커넥터 개념을 이해하고, 가능자는 샘플 연결 또는 도구 연동 흐름을 관찰한다. |
| 검증 | 남에게 설명할 수 있을 만큼 이해한 결과만 받아들인다. | 결과물 구조를 AI에게 설명시키고 본인이 이해한 내용을 다시 쓴다. |
| 컨텍스트 관리 | 대화가 길어지면 현재 상태와 다음 작업만 요약해 새로 이어간다. | 새 대화용 `현재 상태 / 완료 / 다음 작업 / 주의점` 요약문을 작성한다. |
| 에이전트 활용 | 계획, 실행, 검증을 분리해서 AI에게 맡긴다. | 보유자는 계획-실행-검증 루프를 직접 돌리고, 나머지는 화면 공유로 관찰한다. |
| 구조 설계 | 구조와 데이터 모델은 사람이 정하고 AI는 그 안에서 구현하게 한다. | 화면, 데이터, 파일 역할을 사람이 먼저 정한다. |
| 디버깅 | 정상 상태를 기준으로 원인을 좁힌 뒤 수정한다. | 버그나 요구 변경을 넣고 의심 범위를 적은 뒤 수정시킨다. |
| 버전관리 | 기능 단위로 저장 지점을 만들고 문제 시 되돌린다. | git, 히스토리, 복사본 중 하나로 저장 지점을 만든다. |
| 보안 | 비밀값, 개인정보, 계정 권한, 공유 범위를 작업 중 점검한다. | 개인정보, API 키, 계정 권한, 공유 링크 노출을 점검한다. |

## 진행자 체크포인트

- 참가자가 목표를 너무 크게 잡으면 2시간 안에 가능한 첫 산출물로 줄인다.
- 도구 설치 문제로 10분 이상 막히면 웹 기반 경로로 전환한다.
- AI가 만든 결과가 좋아 보여도 반드시 설명, 검증, 보안 메모를 남기게 한다.
- 결과물 종류가 달라도 같은 기준으로 회고한다. 기준은 완성도가 아니라 좋은 AI 협업 루틴을 실제로 거쳤는지다.

## 종료 기준

수업이 끝날 때 각 참가자는 다음을 보여준다.

1. 작게 동작하거나 읽을 수 있는 첫 산출물
2. 목표 분해 기록
3. AI에게 준 핵심 프롬프트
4. 검증한 방법
5. 저장 지점 또는 복구 방법
6. 디버깅 또는 요구 변경 처리 기록
7. 보안 점검 메모
8. 다음 작업을 위한 요약과 위험 메모
