AI CLI로 날씨 API와 DB 앱 만들기
Codex CLI, Claude CLI 계열, Antigravity CLI 계열로 진행하는 크로스체크 실습
오늘의 핵심
사람은 목표, 제약, 검증 기준을 정하고 AI CLI 도구는 계획, 구현, 수정, 검증을 돕습니다.
진행자가 일방적으로 이끄는 방식이 아니라 참가자끼리 계획, 파일 구조, API 응답, DB 저장 결과를 서로 확인합니다.
최종 목표는 날씨 API에서 데이터를 받아오고, SQLite DB에 저장하며, 브라우저에서 조회 결과와 최근 기록을 확인하는 앱을 완성하는 것입니다.
1. 오늘 만들 결과물
각 참가자는 자기 주제나 관심 도시를 바탕으로 날씨 조회 앱을 만듭니다. 앱은 입력, API 호출, DB 저장, 화면 표시가 모두 연결되어야 합니다.
파일/구성요소: 역할
README.md: 프로젝트 목적, 실행 방법, 확인 방법을 설명합니다.
server.js: Node 서버를 실행하고, 날씨 API 호출, SQLite 저장, 최근 조회 기록 조회를 담당합니다.
public/index.html: 브라우저에 보이는 입력창, 버튼, 결과 영역, 최근 기록 영역을 담습니다.
public/style.css: 화면 레이아웃, 색상, 간격, 반응형 표시를 담당합니다.
public/script.js: 사용자 입력을 서버로 보내고, API/DB 결과를 화면에 표시합니다.
TEST_CHECKLIST.md: AI가 만든 기능 체크리스트를 참가자가 수정하고, 크로스체크 결과를 기록합니다.
weather.sqlite: 앱 실행 중 생성되는 SQLite DB 파일입니다. GitHub에는 올리지 않는 파일로 설명합니다.

0:10-0:25 - Node.js와 작업 폴더 확인 - Node/npm 버전, 폴더 위치, 파일 생성 권한을 짝과 확인합니다.

2:30-2:50 - TEST_CHECKLIST.md 기준 상호 검증 - 통과, 수정 필요, 직접 확인 필요 항목을 나눠 기록합니다.

3. 함께 확인하는 실습 원칙
진행자는 정답을 대신 만들어주는 사람이 아니라 흐름과 시간을 관리하는 사람입니다.
참가자는 자기 AI CLI 도구를 사용하되, 같은 체크리스트로 결과를 비교합니다.
AI가 만든 계획은 바로 승인하지 않고 짝과 한 번 읽은 뒤 수정 요청합니다.
API 응답, DB 저장, 화면 표시를 하나로 뭉뚱그리지 않고 각각 확인합니다.
오류가 나면 전체를 다시 만들지 않고 마지막 정상 상태, 방금 바꾼 것, 실제 오류 메시지를 정리합니다.
4. 진행 단계
1단계. AI CLI 도구 선택과 실행 확인 (0:00-0:25)
목표: 각 참가자가 사용할 CLI형 AI 도구를 하나 선택하고 터미널에서 대화 가능한 상태를 만듭니다.
선택 도구 예시는 Codex CLI, Claude CLI/Claude Code 계열, Antigravity CLI 계열입니다.
수업 당일에는 각 도구의 공식 문서에서 설치와 로그인 명령을 확인합니다.
짝과 확인: 서로의 터미널에서 AI CLI가 응답하는지 확인하고, 막힌 사람은 도구 설치보다 계획 작성 흐름을 먼저 따라갑니다.
2단계. 만들 앱을 한 문장으로 정하기 (0:25-0:35)
목표: “도시를 입력하면 현재 날씨를 보여주고 조회 기록을 저장하는 앱”을 기본형으로 잡습니다.
참가자는 관심 도시, 표시하고 싶은 정보, 최근 기록에 남길 항목을 한 문장으로 적습니다.
짝과 확인: 기능이 커지면 지도, 로그인, 배포, 사용자 계정은 제외하고 날씨 조회와 기록 저장만 남깁니다.
3단계. Plan Mode로 앱 구조 계획 세우기 (0:35-1:00)
목표: AI CLI에게 바로 구현을 맡기지 않고 파일 구조, 데이터 흐름, 확인 방법을 먼저 계획하게 합니다.
계획에는 입력값, Open-Meteo 날씨 API 호출, SQLite 저장 테이블, 최근 조회 기록 표시, TEST_CHECKLIST.md가 포함되어야 합니다.
짝과 확인: 다른 참가자의 계획에서 API 호출과 DB 저장이 빠졌는지 확인합니다.
4단계. 계획 크로스체크와 수정 요청 (1:00-1:15)
목표: 각자 받은 계획을 서로 읽고 한 번 이상 수정 요청합니다.
확인 기준은 “입력 → API 요청 → DB 저장 → 화면 표시 → 체크리스트 검증” 흐름이 끊기지 않는지입니다.
계획이 너무 어렵다면 로그인, 외부 지도, 복잡한 배포, 여러 페이지 구조를 빼고 단일 화면 앱으로 줄입니다.
5단계. AI CLI가 앱 구현 (1:15-1:45)
목표: AI CLI가 Node 서버, public 폴더, SQLite 연결, 날씨 API 호출, TEST_CHECKLIST.md를 만들게 합니다.
참가자는 코드를 직접 고치기보다 AI CLI가 만든 파일 목록과 실행 명령을 확인합니다.
짝과 확인: 서로의 폴더에 README.md, server.js, public/index.html, public/style.css, public/script.js, TEST_CHECKLIST.md가 있는지 확인합니다.
6단계. 브라우저에서 날씨 조회 확인 (1:45-2:10)
목표: 로컬 서버를 실행하고 브라우저에서 도시 또는 좌표 입력 후 날씨 결과가 표시되는지 확인합니다.
API 호출이 실패하면 오류 메시지를 지우지 않고 AI CLI에게 원인 후보와 확인 순서를 먼저 묻습니다.
짝과 확인: 한 사람은 입력과 버튼 동작을 보고, 다른 사람은 터미널 오류와 브라우저 화면을 함께 확인합니다.
7단계. SQLite 저장과 최근 기록 확인 (2:10-2:30)
목표: 조회한 날씨 결과가 SQLite DB에 저장되고, 최근 조회 기록이 화면에 다시 표시되는지 확인합니다.
DB 파일은 실행 중 생성될 수 있으며 GitHub에는 올리지 않는 파일로 설명합니다.
짝과 확인: 같은 도시를 두 번 조회했을 때 기록이 추가되는지, 새로고침 후 최근 기록이 유지되는지 확인합니다.
8단계. TEST_CHECKLIST.md 기준 상호 검증 (2:30-2:50)
목표: AI가 만든 기능 체크리스트를 참가자가 수정하고, 항목별 결과를 기록합니다.
체크리스트에는 파일 존재, 서버 실행, API 응답, DB 저장, 최근 기록 표시, 오류 처리, GitHub 제외 파일 확인이 들어가야 합니다.
짝과 확인: 본인 앱은 본인이 설명하고, 체크는 짝이 읽으면서 통과/수정 필요/직접 확인 필요로 표시합니다.
9단계. GitHub 업로드 또는 다음 작업 요약 (2:50-3:00)
목표: 업로드 가능한 사람은 GitHub에 올리고, 시간이 부족한 사람은 다음에 이어갈 상태 요약을 남깁니다.
weather.sqlite, .env, API 키, 개인정보가 업로드되지 않는지 확인합니다.
짝과 확인: GitHub 파일 목록에 소스 파일과 README는 있고 DB 파일은 없는지 확인합니다.

8. 진행자 탈출 조건
AI CLI 설치가 막힌 사람은 짝의 화면을 보며 계획 작성과 체크리스트 검토를 먼저 함께 합니다.
날씨 API가 실패하면 실제 API 연동 대신 샘플 응답 JSON으로 DB 저장 흐름을 먼저 확인합니다.
SQLite 설치나 패키지 오류가 오래 걸리면 DB 저장 단계는 AI CLI가 작성한 설명과 체크리스트 검토까지 진행합니다.
시간이 부족하면 GitHub 업로드보다 API 응답과 DB 저장 확인을 우선합니다.
최소 완성 기준은 브라우저 화면, API 응답 또는 샘플 응답 처리, DB 저장 계획 또는 실제 저장 확인, TEST_CHECKLIST.md 기록입니다.
